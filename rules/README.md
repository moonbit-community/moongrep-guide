# rules

The `rules` package contains the YAML-authored rule sources used by `moongrep`
and the generated MoonBit bundle that applies them at runtime.

## Layout

- `rules/<group>/*.yaml` stores source rules.
- `rules/bundled_rules.mbt` is generated by the bundler. Do not edit it by hand.
- `rules/import.mbt` provides the package-local imports used by generated code.

Rule ids are derived from the YAML path relative to the rules root, with the
`.yaml` or `.yml` suffix removed. For example,
`rules/rabbita/nav-load.yaml` becomes `rabbita/nav-load`.

## Rule YAML schema

Each rule file must contain exactly one YAML document, and the top-level
document must be a mapping.

### Top-level keys

- `package` (required): package path recorded in each `MatchHit`.
- `description` (required): human-readable message attached to each hit.
- `patterns` (required): a non-empty array of pattern objects.

Only these keys are accepted at the top level.

### Pattern keys

- `shape` (required): a MoonBit expression pattern snippet. Identifiers inside
  `shape` only become metavars if they are explicitly declared under
  `metavars`; undeclared identifiers are matched literally.
- `metavars.structural` (optional): metavars bound as raw AST `Json`.
- `metavars.nameonly` (optional): metavars that must normalize to simple
  identifier names before the rule can match.
- `guard` (optional): a non-blank MoonBit expression intended to produce `Bool`.
  When present, it runs after the generated metavar equality checks.

### Metavar rules

- Placeholder-looking names such as `_value` are not special by themselves.
  They must be declared in `metavars` or they will match the literal identifier
  name.
- Declared metavars must actually be used in `shape`.
- A metavar cannot appear in both `structural` and `nameonly`.
- Duplicate names inside the same bucket are rejected.
- Reserved names are `file`, `hits`, `loc`, and any name starting with
  `__moongrep_`.
- Reusing the same structural metavar multiple times in one pattern emits AST
  equality checks in the generated matcher.
- Reusing the same `nameonly` metavar multiple times requires all matched
  occurrences to normalize to the same identifier name.

### Guard semantics

- Structural metavars are passed to guards as `Json`.
- `nameonly` metavars are passed to guards as `String`.
- `nameonly` metavars only work when the matched AST node can be normalized to a
  simple identifier name. Today that includes binder nodes, simple identifier
  expressions, and `Pattern::Var`; if normalization fails, the pattern does not
  emit a hit.
- Guard source is validated during bundling for syntax and non-blank content,
  so syntactically invalid or blank expressions fail fast before code
  generation. Type errors, including guards that do not evaluate to `Bool`, are
  still caught later when compiling the generated bundle.

## Bundling rules

Run the bundler from the module root:

```bash
moon run bundler
```

The bundler defaults are defined in `bundler/config.mbt`:

- rules root: `rules`
- output path: `rules/bundled_rules.mbt`

You can override both positional arguments:

```bash
moon run bundler <rules-root> <output-path>
```

Bundling does the following:

1. Recursively walks the rules root and collects every `.yaml` and `.yml` file.
2. Sorts paths to keep bundle output stable.
3. Parses and validates each rule file.
4. Generates one matcher function per rule plus
   `pub let moongrep_rules_table`.
5. Writes the generated source to `rules/bundled_rules.mbt`.

The bundler will fail when the rules root contains no YAML rule files, or on
invalid YAML shape, unsupported keys, empty `patterns`, duplicate or
overlapping metavar declarations, invalid guard syntax, unused declared
metavars, or generated matcher name collisions.

## Runtime usage

The generated bundle exports:

```moonbit
pub let moongrep_rules_table : Map[String, (String, Json, Array[MatchHit]) -> Bool]
```

`moongrep/apply_bundled_rules.mbt` iterates this table and applies every rule
matcher while traversing AST nodes. To update packaged rules, edit the YAML
files first and rerun `moon run bundler`. Do not edit `bundled_rules.mbt`
manually.
