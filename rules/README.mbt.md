# Writing Rules

The `rules` package stores the YAML-authored rule sources used by `moongrep`
and the generated MoonBit bundle that applies them at runtime.

Start here if you want to add or refine a rule.

See also:

- [RuleSpec.mbt.md](RuleSpec.mbt.md): YAML rule spec with exact AST JSON semantics, accepted keys, and failure cases

## What Lives Here

- `rules/<group>/**/*.yaml` and `rules/<group>/**/*.yml` are rule source files.
- `rules/bundled_rules.mbt` is generated by `moon run bundler`.
- `rules/import.mbt` provides package-local imports used by generated code.

Rule ids come from the rule file path relative to the rules root, with the
suffix removed. For example, `rules/rabbita/nav-load.yaml` becomes
`rabbita/nav-load`.

## Mental Model

Write rules against the shape of a single AST subtree, not against a whole file.

`moongrep` parses a source file, walks each top-level impl, and tries every
rule pattern against every visited subtree. A rule emits a hit when one of its
`patterns` matches the current subtree.

That means:

- `shape` should be the smallest expression snippet that captures the thing you
  want to flag.
- Multiple entries under `patterns` are OR-ed together.
- All matched patterns in one rule share the same `package` and `description`.
- `guard` is for extra logic after the structural shape already matched.

If you need the exact matcher semantics, read [RuleSpec.mbt.md](RuleSpec.mbt.md).

## Workflow

### 1. Pick the smallest valid `shape`

Start from a concrete source snippet that already looks like the code you want
to flag, then shrink it until only the essential structure remains.

Good starting shapes:

```yaml
patterns:
  - shape: _conn.read_request()
```

```yaml
patterns:
  - shape: |
      for counter = _start; counter < upper_limit; counter = counter + 1 {
        body
      }
```

`shape` is parsed as a single MoonBit expression snippet. If you paste a whole
file or a fragment that only makes sense at module scope, bundling will fail.

### 2. Decide which names are literal and which are metavars

Names in `shape` are literal by default, even if they look like placeholders.

This:

```yaml
patterns:
  - shape: _expr == _expr
```

does **not** create a metavariable. It matches the literal identifier name
`_expr` on both sides.

To make it a metavar, declare it explicitly:

```yaml
patterns:
  - shape: _expr == _expr
    metavars:
      subtree: [_expr]
```

The built-in exceptions are names made only of two or more underscores, such as
`__`, `___`, and `____`. They are special ignore placeholders inside `shape`,
so you must not declare them under `metavars`.

```yaml
patterns:
  - shape: foo(__)
```

When one of these names appears in a supported metavar position, it matches
anything there without binding a guard parameter or participating in
repeated-name equality. In unsupported positions such as method names, it stays
literal. This rule is separate from MoonBit's own special handling of `_`.

If bundling says a declared metavar is unused, the usual causes are:

- a typo between `shape` and `metavars`
- the name only appears in a position that does not support metavariable
  binding

The exact supported positions are listed in [RuleSpec.mbt.md](RuleSpec.mbt.md).

### 3. Choose `subtree` or `identifier`

Use `subtree` when you want the full matched AST subtree as `Json`, or when
repeating a metavar should mean "the subtree is structurally identical,
ignoring source locations".

```yaml
patterns:
  - shape: _expr == _expr
    metavars:
      subtree: [_expr]
```

This is a good fit for matches such as:

- `x == x`
- `items[i] == items[i]`
- `user.profile.name == user.profile.name`

Use `identifier` when you want to compare identifier spelling across positions
that are not the same raw AST node kind, especially binder positions versus
identifier uses.

```yaml
patterns:
  - shape: |
      for counter = _start; counter < upper_limit; counter = counter + 1 {
        body
      }
    metavars:
      subtree: [_start, upper_limit, body]
      identifier: [counter]
```

Here `counter` appears as both a binder and later identifier expressions. The
same spelling should match, but the raw AST nodes are different, so
`identifier` is the right tool.

`identifier` also works for simple assignment targets represented in the AST as
`Var`, so rules like `x = x + 1` can bind the left-hand target by normalized
name instead of treating it as a literal string.

### 4. Add a `guard` only for logic shape matching cannot express

Reach for a guard when the structure alone is not enough.

Typical uses:

- "the loop variable is not used in the body"
- "the same captured subtree passes an extra helper predicate"
- "the matched name has to satisfy a string-level property"

Example:

```yaml
package: moonbitlang/core
description: |
  This C-style `for` loop updates a loop variable that the body never reads.
patterns:
  - shape: |
      for counter = _start; counter < upper_limit; counter = counter + 1 {
        body
      }
    metavars:
      subtree: [_start, upper_limit, body]
      identifier: [counter]
    guard: |
      @greptools.is_unused_var(upper_limit, counter) &&
      @greptools.is_unused_var(body, counter)
```

Write the simplest guard that works. If the same thing can be expressed by a
more specific `shape`, prefer the more specific shape.

### 5. Add more `patterns` when the message is shared

If several surface forms deserve the same rule id, package, and description,
put them in one rule file.

```yaml
package: moonbitlang/async/http
description: |
  These HTTP parser entrypoints accept messages where `Content-Length` and
  `Transfer-Encoding` may coexist.
patterns:
  - shape: |
      _conn.read_request()
    metavars:
      subtree: [_conn]
  - shape: |
      _client.end_request()
    metavars:
      subtree: [_client]
```

Use separate rule files only when the rule id, message, or ownership should be
different.

### 6. Bundle and inspect the result

Run the bundler from the module root:

```bash
moon run bundler
```

The defaults are:

- rules root: `rules`
- output path: `rules/bundled_rules.mbt`

You can also override both positional arguments:

```bash
moon run bundler <rules-root> <output-path>
```

To inspect the CLI contract directly:

```bash
moon run bundler --help
```

## Worked Examples

### Repeated subtree equality

```yaml
package: moonbitlang/core
description: |
  Repeated subtree equality.
patterns:
  - shape: _expr == _expr
    metavars:
      subtree: [_expr]
```

Why it works:

- `_expr` is declared as a subtree metavar
- both occurrences must bind to structurally equal AST JSON
- the rule can match more than simple names

### Binder and use must share one source-level name

```yaml
package: moonbitlang/core
description: |
  Counter-style `for` loop.
patterns:
  - shape: |
      for counter = _start; counter < upper_limit; counter = counter + 1 {
        body
      }
    metavars:
      subtree: [_start, upper_limit, body]
      identifier: [counter]
```

Why it works:

- `counter` is compared by normalized identifier name, not by raw AST equality
- `_start`, `upper_limit`, and `body` stay available to guards as `Json`

### Same rule, multiple shapes

```yaml
package: moonbitlang/async/process
description: |
  These helpers collect full child-process output into memory before returning.
patterns:
  - shape: _command.output_collect(_args)
    metavars:
      subtree: [_command, _args]
  - shape: _command.stderr_collect(_args)
    metavars:
      subtree: [_command, _args]
```

Why it works:

- each pattern is a separate OR arm
- either arm emits the same rule metadata
- the generated hit records which arm matched through `pattern_index`

## Debugging Checklist

### The bundler says `shape` is invalid

Your snippet is not accepted by the expression parser. Reduce it to one valid
expression-sized shape, then build back up carefully.

### The bundler says a metavar is unused

Check, in order:

- the spelling matches exactly between `shape` and `metavars`
- the name appears in a metavariable-capable position
- you declared it in the correct bucket

### An `identifier` rule looks right but never hits

The matched node may not normalize to a simple identifier name in every
occurrence. Review the exact supported normalization cases in
[RuleSpec.mbt.md](RuleSpec.mbt.md).

### The guard fails

There are two distinct failure modes:

- syntax failure during bundling: the guard is not valid MoonBit expression
  syntax
- compile failure after code generation: the guard parses, but does not type
  check as generated code

## Testing Workflow

After changing a rule:

1. rerun `moon run bundler`
2. add or update a focused wbtest for the generated matcher
3. cover both a positive case and at least one nearby negative case

[rules/language_wbtest.mbt](language_wbtest.mbt) is the model to follow. It shows the
recommended pattern:

- create an in-memory source string
- traverse it with `@greptools.traverse_top_impls(...)`
- call the generated matcher
- assert on the resulting `MatchHit`s

Keep tests narrow. A good rule test proves the intended match and at least one
non-match that would be easy to regress.
